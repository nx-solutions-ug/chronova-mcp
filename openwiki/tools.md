# MCP Tools

All four tools are registered in `src/server.ts` and `src/stdio.ts` via `McpServer.registerTool` and marked `readOnlyHint: true`. Each handler follows the same pattern:

1. Build the upstream path + query string
2. Call `chronova.get<T>(...)` 
3. Return the response `data` as a single JSON-stringified `text` content block
4. On `ChronovaApiError`, return the error message as `text` with `isError: true`

If you add a new tool, register it in **both** `server.ts` and `stdio.ts` — the HTTP and stdio entry points keep their own `McpServer` instances.

## `get_developer_context`

- **File:** `src/tools/get-developer-context.ts`
- **Parameters:** none
- **Upstream:** `GET users/current`
- **Response type:** `ChronovaUser` (see `src/lib/types.ts`)

Returns the authenticated developer's profile, including:

- `id`, `username`, `email`, `avatar_url`
- `subscription` (`plan`, `status`)
- `github_connected` boolean
- `organizations` array (`id`, `name`, `role`)
- `created_at`, `modified_at` timestamps

Useful for an agent that wants to know who it is talking to before showing stats or filtering by project.

## `get_productivity_summary`

- **File:** `src/tools/get-productivity-summary.ts`
- **Parameters:**
  - `range` (required) — enum: `today`, `last_7_days`, `last_30_days`, `last_3_months`, `last_6_months`, `last_year`, `all_time`
  - `project` (optional) — string, filter to a specific project
- **Upstream:** `GET users/current/stats/{range}` (with `?project=…` if provided)
- **Response type:** `ChronovaStatsRange`

Returns aggregated coding-time stats for the range:

- `total_seconds`
- `languages`, `projects`, `editors`, `operating_systems` — each `[{ name, total_seconds, percent }]`
- `daily_stats`, `hourly_stats` — for trend visualization
- `best_day` (`{ date, total_seconds }` or `null`)
- `start`, `end` — actual date range covered

## `get_ai_insights`

- **File:** `src/tools/get-ai-insights.ts`
- **Parameters:**
  - `range` (required) — same enum as `get_productivity_summary`, **plus** custom `YYYY-MM-DD_to_YYYY-MM-DD` strings (the schema accepts both via `z.union([z.enum([...]), z.string().regex(...)])`)
- **Upstream:** `GET users/current/analytics/ai?range=…`
- **Response type:** `ChronovaAiAnalytics`

Returns AI-assisted coding analytics:

- `adoptionTimeline` — daily `aiSeconds` / `manualSeconds`
- `contributionShare` — `aiPercent` / `manualPercent`, plus hour totals
- `comparison` — `withAi` vs `withoutAi` (`totalSeconds`, `avgDaily`)
- `languageMatrix` — per-language AI vs manual share
- `projectDependency` — per-project AI vs manual share
- `efficiencyTrend` — `[{ period, productivity }]`

## `get_recent_activity`

- **File:** `src/tools/get-recent-activity.ts`
- **Parameters (all optional):**
  - `date` — `YYYY-MM-DD` (single day)
  - `start`, `end` — `YYYY-MM-DD` (range)
  - `project` — project name
  - `language` — programming language
  - `editor` — editor/IDE name
  - `page` — positive int, default 1
  - `per_page` — positive int, default 100, max 100
- **Upstream:** `GET users/current/heartbeats?<filters>`
- **Response type:** `ChronovaHeartbeatResponse`

Returns paginated raw heartbeats. Each heartbeat has `id`, `time`, `type`, `project`, `language`, `editor`, `operating_system`, `machine`, `branch`, `created_at`. The response includes `total`, `page`, `per_page`, `total_pages` so an agent can paginate.

Note: only defined filter values are sent — empty strings and `undefined` are dropped by the tool handler.

## Adding a new tool

1. Add a new file in `src/tools/your-tool.ts` exporting a `registerXxx(server, chronova)` function.
2. Import and call the registrar from `createMcpServer` in `src/server.ts` and from `main()` in `src/stdio.ts`.
3. Add a new type to `src/lib/types.ts` if the upstream payload is non-trivial.
4. Add tests in `tests/integration/tools.test.ts` using the `mockChronovaApi()` helper (see `testing.md`).
5. Update `README.md` and `tools.md` (here) with the new tool's parameters.

## Error responses

Every tool returns MCP errors as `text` content with `isError: true`. The text is generated by `mapHttpStatusToError` (see `api-client-and-errors.md`) and typically tells the agent what to fix (e.g. `CHRONOVA_API_KEY` guidance for 401s, `Retry-After` for 429s, `CHRONOVA_API_URL` guidance for network errors).
