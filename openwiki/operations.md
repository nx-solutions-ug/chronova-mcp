# Operations

Build, release, and CI workflows for the Chronova MCP server.

## Local development

```bash
npm install
npm run dev          # tsc --watch & node --watch dist/index.js
npm test             # vitest run
npm run type-check   # tsc --noEmit
npm run lint         # eslint .
npm run build        # tsc -> dist/
```

`package.json` has a `prepublishOnly: "npm run build"` hook so the npm tarball only contains compiled output (`dist/`, `README.md`).

## Docker

`Dockerfile` is a two-stage Node 24 Alpine build:

1. **Builder stage** â€” installs full `devDependencies`, copies `src/`, runs `tsc`
2. **Runtime stage** â€” copies `dist/`, `node_modules/`, and `package.json`, exposes `PORT=3001`, runs `node dist/index.js`

```bash
docker build -t chronova-mcp .
docker run -e CHRONOVA_API_KEY=your-key -p 3001:3001 chronova-mcp
```

## Release

`.github/workflows/release.yml` runs on push to `main`:

1. **test job** â€” runs `npm ci`, then `npm run type-check` and `npm run lint`
2. **release job** â€” uses a GitHub App token (via `actions/create-github-app-token@v3`) and `npx semantic-release` to publish to npm and cut a GitHub release

`.releaserc.json` configures semantic-release to:

- Track `main`, `beta` (prerelease), `alpha` (prerelease)
- Update `CHANGELOG.md`, `package.json`, `package-lock.json` in the release commit
- Publish to npm with `@semantic-release/npm`
- Create a GitHub release (notes truncated to 120 KB with a link to the full changelog)
- Tag format: `v${version}`

The `CHANGELOG.md` is regenerated by the changelog plugin on every release; do not edit it by hand.

## Continuous integration

`.github/workflows/test.yml` runs on push and PR to `main`, `develop`, `feat/*`, `fix/*`:

| Job | What it does |
|---|---|
| `type-check` | `npm run type-check` |
| `lint` | `npm run lint` |
| `build` | `npm run build` |
| `test` | `npm test` |

Concurrency is grouped by workflow + ref and cancels in-flight runs on the same ref.

## OpenWiki automation

`.github/workflows/openwiki-update.yml` keeps the `openwiki/` directory current:

- **Triggers:** `workflow_dispatch`, push to `main`, and a daily schedule (`0 8 * * *` UTC)
- Builds the OpenWiki CLI from the `feat/ollama-provider` branch of `niklasschaeffer/openwiki`
- Runs `node /tmp/openwiki/dist/cli.js --update --print` with Ollama + LangSmith env vars from secrets
- Opens (or updates) a PR with `add-paths: openwiki`, branch `openwiki/update`, using a GitHub App token

If you change source code, expect the next scheduled run to open an OpenWiki PR that needs review before merge.

## OMP / issue-management workflows

These are repository-internal automation (Outbound Maintenance Processor) and not part of the MCP server itself. They live in `.github/workflows/`:

- `auto-manage.yml` â€” auto-assigns issues/PRs and applies labels
- `omp.yml` / `omp-ci.yml` / `omp-fix-issue.yml` â€” issue triage and fixes via the `chronova-agent` app

They consume the `.omp/` configuration (commands, rules, agent config) and `.omp/stream-log.py`. You generally do not need to touch these to work on the MCP server.
